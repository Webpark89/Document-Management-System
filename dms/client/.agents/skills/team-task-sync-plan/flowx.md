# แนวทางการทำงานแบบ Frontend-First "Soft Create" 🎨

เอกสารนี้ใช้เป็น **"คู่มือหรือกฎประจำใจ"** ก่อนเริ่มเขียนโค้ดในโปรเจกต์นี้ เพื่อให้มั่นใจว่าการทำงานจะโฟกัสที่ Frontend 100% โดย **ไม่เข้าไปยุ่งเกี่ยวกับ Backend (NestJS, Database) โดยเด็ดขาด** จนกว่าจะได้รับคำสั่ง

---

## 🎯 หัวใจหลักของการทำ "Soft Create"
คือการสร้าง **UI ที่มีชีวิตชีวาและใช้งานได้จริง (Interactive UI)** ระบบต้องตอบสนองต่อผู้ใช้ได้สมบูรณ์แบบเสมือนต่อ Database อยู่จริงๆ เพียงแต่ข้อมูลทั้งหมดจะถูกเก็บไว้ในหน่วยความจำชั่วคราว (Mock Data / Client State)

### 🚫 กฎเหล็ก 3 ข้อ (Strict Rules)
1. **ห้ามยุ่งกับ Backend:** ห้ามสร้างไฟล์ในฝั่ง NestJS, ห้ามแก้ `schema.prisma`, และห้ามพูดถึง Database
2. **ห้ามเรียก API จริง:** ห้ามใช้ `axios.get` หรือ `fetch` ไปยัง Endpoint จริงๆ
3. **ต้องสมจริง:** ทุกปุ่มต้องกดได้ มี Loading State มีการแจ้งเตือน (Toast/Alert) และข้อมูลบนหน้าจอต้องเปลี่ยนไปตาม Action นั้นจริงๆ

## 🛠️ ลักษณะการเขียนโค้ด (Implementation Guide)

เพื่อให้เป็นไปตามกฎเหล็ก เวลา AI หรือ Developer สร้างฟีเจอร์ใหม่ ให้ใช้เทคนิคต่อไปนี้:

### 1. การจัดการข้อมูล (Data Mocking & Volatile State)
- **สร้างไฟล์ API ปลอม:** เก็บข้อมูลสมมติ (Mock Data) ไว้ในไฟล์เช่น `src/features/[feature]/api.ts`
- **Volatile State (จำลองแบบมีชีวิต):** ตามที่พี่เลี้ยงแนะนำ ต้องทำให้ระบบสามารถกดเพิ่ม/ลบ ข้อมูลได้จริง โดยให้เขียนโค้ดอัปเดตตัวแปร Array ชั่วคราวในฝั่ง Frontend
  - **ตัวอย่าง:** เมื่อกดปุ่ม "เพิ่มเอกสาร" ข้อมูลใหม่จะต้องไปโผล่ในหน้า "ตารางเอกสาร" ทันที 
  - **ข้อควรจำ:** เนื่องจากเป็นการเก็บข้อมูลชั่วคราว (In-Memory) เมื่อกดรีเฟรชหน้าเว็บ (F5) ข้อมูลที่เพิ่งเพิ่มเข้าไปจะหายไปและกลับไปสู่ค่าตั้งต้น ถือว่าเป็นเรื่องปกติของการทำ Soft Create
- **ใช้ข้อมูลสมจริง:** เขียนข้อมูล Mock ให้มีความหมายและตรงกับบริบททางธุรกิจ (เช่น ชื่อคนไทย, วันที่ที่สมเหตุสมผล, ตัวเลขเงินบาท)
- **จำลอง Network Delay:** ทุกฟังก์ชันที่ทำหน้าที่เป็น API จะต้องใส่การหน่วงเวลาเพื่อจำลองความสมจริง
  ```typescript
  // ตัวอย่างการเก็บ State ชั่วคราวใน Memory
  let MOCK_DOCUMENTS = [ ... ];
  
  export async function addDocument(newDoc) {
    await new Promise(resolve => setTimeout(resolve, 500));
    MOCK_DOCUMENTS.push(newDoc); // เพิ่มเข้า Memory ชั่วคราว
  }
  ```

### 2. การจัดการสถานะ (State & Interactions)
- **Client-Side State:** ใช้ `useState`, `useReducer`, หรือ Zustand ในการจัดการข้อมูลที่เปลี่ยนไปเมื่อผู้ใช้โต้ตอบ (เช่น กดเพิ่มแถวในฟอร์ม, พิมพ์ข้อความค้นหา)
- **URL Parameters:** หากเป็นหน้าที่ต้องการการแชร์ลิงก์ (เช่น การกรองข้อมูล, เลขหน้า) ให้ดึงค่าผ่าน `searchParams`

### 3. การจำลองการบันทึกข้อมูล (Simulating Mutations)
เมื่อผู้ใช้กดปุ่ม Submit ห้ามทำแค่ Alert แล้วจบ ต้องทำให้สมบูรณ์:
1. เปลี่ยนปุ่มเป็น **สถานะกำลังโหลด (Loading Spinner)**
2. เรียกฟังก์ชัน Mock API เพื่อ **หน่วงเวลา**
3. **อัปเดตข้อมูลชั่วคราว** ใน State บนหน้าจอให้เห็นว่าสำเร็จ
4. เด้งหน้าต่างแจ้งเตือน (Success Notification)
5. เปลี่ยนหน้า (Redirect) ไปยังหน้าที่เหมาะสม

### 4. การจัดการ Auth (Mock Session)
- สมมติสถานะการล็อกอินเอาไว้ใน Frontend เลย (เช่น เก็บ User Object สมมติไว้ใน Context หรือ LocalStorage)
- ทำ UI ให้เปลี่ยนไปตามสิทธิ์ของ Role ปลอมๆ ที่เราตั้งค่าไว้ (เช่น สิทธิ์ Admin เห็นปุ่มตั้งค่า, สิทธิ์ User ธรรมดาไม่เห็น)
